WMPP 架构与论文重点说明

本页用于论文与答辩说明：严格区分 WMPP 推送基础设施 与 业务系统。业务系统只接入 SDK；WMPP 负责高并发推送、存储转发、负载均衡、监控统计与安全隔离。

系统边界清晰 高并发 负载均衡 多租户隔离 可观测性

1. 系统分层（论文主体）

业务系统层（电商/社区/短视频）
Server：引入 SDK 发起推送请求；Client：登录上线后被动接收。
WMPP 网关与鉴权层
统一入口 API、appId/appSecret 验证、租户隔离校验。
推送调度层（Scheduler）
按策略分配到 pusher 实例：轮询 / 最小连接数。
推送执行层（Pusher Pool）
多个实例横向扩展，承载 WebSocket/SSE 长连接与消息下发。

2. 高并发与扩展策略

  • 基础能力：多 pusher 实例 + 负载均衡，避免单点瓶颈。
  • 扩展能力：支持自动伸缩策略（阈值配置），按负载动态调整实例数。
  • 削峰能力（可扩展）：引入消息队列（Kafka/RabbitMQ）实现异步缓冲。
  • 故障隔离（可扩展）：断路器/熔断，单实例故障不影响整体。

容器化建议

每个 pusher 对应一个容器实例，可动态启动/停止，实现资源按需分配。

3. 消息存储与转发（按场景选型）

  • MySQL：业务配置、租户信息、推送任务元数据。
  • Redis：在线会话、热点路由、短期状态缓存。
  • MongoDB（可选）：灵活结构的消息记录与回执数据。
  • 对象存储 MinIO（可选）：图片/文件等二进制消息体。

消息可按“活动期保留、过期清理”策略管理，降低长期存储成本。

4. 监控统计与可视化

  • 业务系统侧：查询“已发送/已送达/失败数”等本租户统计。
  • 平台管理员侧：查看所有租户总量、并发趋势、实例负载。
  • 论文可给出压力测试：单实例 vs 多实例 的响应时间与吞吐对比。

6. 论文可直接引用的核心结论

WMPP 通过“配置模块与推送模块分离、pusher 池横向扩展、基础负载均衡策略、租户级鉴权隔离、可观测统计接口”实现了面向多业务系统的可扩展消息推送基础设施。业务系统仅需引入 SDK 即可完成接入，客户端形态和业务语言不受限制。

关键词：Microservice / Load Balancer / Pusher Pool / Multi-tenant Isolation / Observability
