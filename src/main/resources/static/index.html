一、页面入口

业务系统A - 消息推送设置页面（systemA / 电子书网站） 只支持单次推送；用于展示“业务系统 Server 接入 SDK 后完成一次推送”的标准流程。 业务系统B - 消息推送设置页面（systemB / 电商网站） 支持广播、指定用户、指定用户组以及周期推送；用于展示多策略混合能力。 业务系统C - 消息推送设置页面（systemC / 音乐网站） 支持广播、定向与周期推送，强调跨终端一致接收能力。 业务系统 Client 接收页（systemA / 1001） 模拟业务系统用户登录上线后接入 client sdk，只负责被动接收平台推送消息。 业务系统 Client 接收页（systemA / 2002） 用于演示同一业务系统下多个客户端同时在线并同时收到消息。 业务系统 Client 接收页（systemB / 3003） 用于演示不同业务系统之间的数据隔离与消息隔离。 业务系统 Client 接收页（systemC / 5001） 用于演示短视频业务系统客户端登录上线后，被动接收 systemC 消息。 租户控制台（客户后台） 客户登录后只能看自己的租户数据、自己的在线终端、自己的 SDK 配置和自己的路由，不可见其他客户数据。 Admin 控制台（平台总后台） 平台管理员查看全部租户、全部在线数据、节点承载、全量路由以及负载均衡演示区。 WMPP 架构与高并发说明（论文重点页） 集中展示微服务拆分、负载均衡、pusher 池、消息存储、监控统计、可扩展能力与论文可写点。

二、推荐答辩打开顺序

1. 先说明：WMPP 本身不是业务系统，而是独立消息推送平台；业务系统只需要在 Server 端引入 SDK（new 对象 + set 配置 + push），在 Client 端登录后 connect 并被动接收。
2. 打开 /biz-server-demo.html，讲“业务系统1 的后端如何引入 server sdk 并配置推送规则”。
3. 再打开多个 Client 页：systemA/1001、systemA/2002、systemB/3003，讲“终端一登录就只负责接收消息”。
4. 在业务系统1 页面做广播、固定用户推送、一组用户推送，观察不同 Client 的接收差异。
5. 再打开 /biz-server-demo-2.html，说明这是另一个业务系统；虽然调用 SDK 方式完全一样，但由于 appId 不同，消息不会发送到 systemA。
6. 最后展示 /tenant-console.html 和 /admin-console.html，说明租户隔离、全局监控、高并发与负载均衡能力。
